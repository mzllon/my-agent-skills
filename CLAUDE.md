## Document Update Rules

每次任务完成后：
1. 检查现有文档是否需要更新（新 API、新架构决策、新约定）
2. 同一类错误犯了第二次 → 补充到 `docs/lessons.md`
3. 新增了 docs/ 文档 → 在本文件索引表中加一行
4. 本文件不放任何实质内容，所有内容在 docs/ 对应文件中维护

## 交互规范

### 需要用户确认的决策必须逐个、交互式提问

- 凡是需要用户拍板的事项（命名、路径、依赖版本、字段类型、DDL 策略等），**必须使用交互式提问工具（AskUserQuestion）逐个单独发起**，一次只问一个问题，并**等待用户确认后再继续**。
- **禁止**把多个待确认项以普通文字罗列在工作总结、变更清单或报告的末尾（例如「以上已完成，请确认 A / B / C 三点」）。
  - 原因：正文过长时，结尾的咨询文字会被滚动顶没，用户容易漏答或直接错过，导致流程卡住或基于错误假设推进。
- 也**禁止**在长篇内容中间穿插咨询用户——确认必须放在一轮输出的结尾，且只问当前最关键、阻塞性的那一个问题，保持视线聚焦。
- 提问时给出明确的默认/推荐选项（标注「(Recommended)」），但最终以用户选择为准。
- 拿到确认前不要假设答案并继续；用户未答复就视为未决策。
